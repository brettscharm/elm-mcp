# Agentic Development Lifecycle with IBM Bob + ELM

> **Art of the Possible: AI-Driven Engineering with Full Audit Trail**

This document describes the complete agentic development lifecycle using IBM Bob (AI coding assistant) integrated with IBM Engineering Lifecycle Management (ELM). Every artifact Bob creates is tagged, traceable, and governed by human approval gates.

---

## The Vision

Bob acts as an AI engineering agent that can:
- **Generate** requirements, work items, and test cases
- **Read** existing artifacts across DNG, EWM, and ETM
- **Link** everything together for full traceability
- **Implement** code from approved requirements
- **Test** against the test plan
- **Update** ELM artifacts as work progresses

The human stays in control at every gate. ELM provides the audit trail.

---

## Full Lifecycle

### Phase 1: Requirements (DNG) ✅ BUILT

**What Bob does:**
1. User describes a system or feature
2. Bob interviews the user (what type, how many, constraints, links)
3. Bob generates requirements with acceptance criteria
4. User previews in a clean table, requests changes if needed
5. Bob pushes to DNG in an `AI Generated - user - summary` folder
6. Bob creates links between requirements (Satisfies, Elaborated By, etc.)

**Human gate:** Review requirements in DNG → move to module → set status to Approved

**ELM artifacts created:**
- Requirements with `[AI Generated]` prefix and `[AI Generated by Bob]` in content
- Folders for organization
- Links between requirements

**Status:** ✅ Fully working — tested against live DNG server

---

### Phase 2: Development Planning (EWM) ✅ BUILT

**What Bob does:**
1. Bob reads ONLY Approved requirements from DNG
2. If any are not Approved, Bob warns: "These requirements aren't approved yet — proceed anyway?"
3. Bob generates Tasks with acceptance criteria derived from requirements
4. User previews, confirms → Bob pushes to EWM with `[AI Generated]` prefix
5. Each work item linked to its source requirement (`Implements Requirement`)

**Human gate:** Assign work items to iterations → prioritize → assign to developers

**ELM artifacts created:**
- Tasks in EWM (via `create_task` MCP tool)
- Cross-tool links: EWM Task → DNG requirement (calm:implementsRequirement)

**Status:**
- ✅ EWM project discovery works (101 projects found)
- ✅ EWM creation factory discovery (Task, Defect, Story factories)
- ✅ Task creation via MCP tool — tested against live server (201 Created)
- ✅ Cross-tool link: Task → DNG Requirement (calm:implementsRequirement)
- ✅ OSLC error messages surfaced to user (permissions, etc.)
- 🔧 Story/Defect creation skipped for now (Filed Against namespace issue)

---

### Phase 3: Test Planning & Execution (ETM) ✅ BUILT

**What Bob does:**
1. Bob reads Approved requirements from DNG
2. Bob generates Test Cases with test steps derived from acceptance criteria
3. Bob generates Test Scripts (manual or automated)
4. User previews, confirms → Bob pushes to ETM with `[AI Generated]` prefix
5. Each test case linked to its source requirement (`oslc_qm:validatesRequirement`)
6. Bob creates Test Execution Records and records Test Results with pass/fail

**Human gate:** Review test plan → approve test cases → review test results

**ELM artifacts created:**
- Test Cases in ETM
- Test Scripts in ETM
- Test Execution Records in ETM
- Test Results (pass/fail) in ETM
- Cross-tool links: ETM test case → DNG requirement

**Status:**
- ✅ ETM project discovery works (77 projects found)
- ✅ Test Case creation works
- ✅ Test Script creation works
- ✅ Test Execution Record creation works
- ✅ Test Result creation with pass/fail works
- ✅ Cross-tool link: Test Case → DNG Requirement (oslc_qm:validatesRequirement)

---

### Phase 4: Implementation

**What Bob does:**
1. Bob reads an assigned Story/Task from EWM
2. Bob pulls the linked DNG requirement for full context
3. Bob writes code based on requirements + acceptance criteria
4. Bob writes unit tests alongside code
5. Bob updates EWM work item: `New → In Progress → Resolved`
6. Bob commits code to Git

**Human gate:** Code review → approve merge

**ELM artifacts updated:**
- Work item status changes tracked in EWM
- Git commits linked to work items (if EWM-Git integration configured)

**Status:** 🔜 Bob already writes code natively — need EWM status update API

---

### Phase 5: Test Execution

**What Bob does:**
1. Bob runs test cases from ETM against implemented code
2. **Tests pass** → Bob records Pass result in ETM, marks test as Passed
3. **Tests fail** → Bob creates a Defect in EWM linked to:
   - The failing test case
   - The source requirement
   - The implementing story
4. Defect goes through its own lifecycle (assigned → fixed → retested)

**Human gate:** Review test results → approve release

**ELM artifacts updated:**
- Test Execution Records in ETM
- Test Results (Pass/Fail)
- Defects in EWM (if tests fail)

**Status:** ✅ Test Execution Records and Test Results (pass/fail) confirmed working in ETM

---

### Phase 6: Completion & Audit Trail

When all phases complete, ELM contains the full traceable chain:

```
Requirement (DNG)
  ├── Status: Approved
  ├── Satisfies → Parent Requirement
  ├── Implemented By → Story (EWM)
  │     ├── Status: Closed
  │     ├── Has Child → Task (EWM)
  │     └── Tested By → Test Case (ETM)
  │           ├── Status: Passed
  │           └── Test Result → Execution Record
  └── Validated By → Test Case (ETM)
```

Every artifact is:
- Tagged with `[AI Generated]` so humans know it came from Bob
- Timestamped with creation date
- Linked to its source artifacts for full traceability
- Governed by human approval gates at every phase boundary

## Safety Guardrails

| Rule | Why |
|------|-----|
| `[AI Generated]` prefix on all titles | Humans can instantly identify AI work |
| `[AI Generated by Bob]` in content body | Visible when artifact is opened |
| Status check before downstream work | Never develop against unapproved requirements |
| Human approval gates between phases | Human stays in control |
| Never modify existing artifacts | Only create new ones |
| Never touch Approved requirements | Prevents overwriting human decisions |
| Preview before every write | User sees exactly what will be created |
| Descriptive folders per session | Easy to find and review AI output |
| Non-approved source note | Tracks if work was derived from draft requirements |

## Architecture

```
User (VS Code)
  │
  ▼
Bob (AI Assistant)
  │
  ▼
MCP Server (doors_mcp_server.py)
  │
  ▼
ELM Client (doors_client.py)
  │
  ├── /rm  → DOORS Next (DNG) — Requirements
  ├── /ccm → Engineering Workflow (EWM) — Work Items
  └── /qm  → Engineering Test (ETM) — Test Cases
```

All three tools share the same base URL and authentication.
Same OSLC protocol, same session, same credentials.
